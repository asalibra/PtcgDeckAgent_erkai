# PTCG Deck Agent 动作归属矩阵

## 目的

本表用于明确网络对战迁移到 Windows / Android 原生客户端后，哪些动作必须继续由服务器权威裁定，哪些动作只在客户端做预校验或交互辅助。

原则：

- 会改变对局真实状态、隐藏信息、随机结果、胜负结论的动作，必须由服务器最终裁定。
- 客户端可以做本地按钮可用性判断、参数完整性检查、目标选择引导、动画预演，但不能把这些预判直接当成最终事实。
- 一旦客户端本地预判与服务器裁定不一致，以服务器为准，并触发 UI 回滚或重同步。

## 房间与会话消息

| 消息 / 动作 | 客户端可做 | 服务端必须做 | 备注 |
| --- | --- | --- | --- |
| `create_room` | 房间名非空、字符长度检查 | 创建房间、分配 `room_id`、分配 `session_token` | 服务器返回唯一房间标识 |
| `join_room` | 房间 ID 格式检查 | 校验房间存在、是否已满、加入房间 | 不能信任客户端传的加入资格 |
| `list_rooms` | 无 | 枚举当前可加入房间 | 服务器是唯一真实来源 |
| `select_deck` | 本地牌组完整性检查 | 校验牌组存在、可加载、可用于房间 | 服务器仍要验证牌组数据 |
| `set_ready` | 本地 UI 切换按钮状态 | 记录准备状态、通知对手 | 准备状态以服务器为准 |
| `start_game` | 房主按钮是否可点、双方是否已准备的本地提示 | 最终校验房主身份、双方 ready、构建对局 | 只能由服务器开始对局 |
| `reconnect` | 自动重连、展示恢复中状态 | 校验 `session_token`、恢复 peer、下发快照 | 重连恢复必须服务端裁定 |
| `leave_room` | 本地确认弹窗 | 真正离房、通知对手、清理房间引用 | 不能只在客户端本地离开 |

## 对局动作

| 动作 | 客户端预校验 | 服务端权威裁定 | 说明 |
| --- | --- | --- | --- |
| `setup_place_active` | 手牌里是否存在该基础宝可梦 | 是否允许放置、真正修改开局状态 | 开局属于真实状态变更 |
| `setup_place_bench` | 手牌里是否存在该基础宝可梦 | 是否允许继续放置到备战区 | 服务端决定最终备战区内容 |
| `setup_complete` | 当前客户端是否已经完成摆放 | 是否双方都完成 setup，是否进入下一阶段 | 推进阶段必须后端做 |
| `play_basic_to_bench` | 卡牌类型是否为基础宝可梦、目标是否有空位 | 是否满足规则、真正从手牌进入备战区 | 真实状态变更 |
| `evolve` | 进化对象与目标槽位参数是否齐全 | 是否满足进化条件、是否真的进化成功 | 不能只信客户端的合法性判断 |
| `attach_energy` | 源卡是否在手牌、目标槽位是否存在 | 每回合次数限制、特殊效果修正、真正附能 | 会影响后续所有动作合法性 |
| `attach_tool` | 目标参数是否完整 | 是否能装备、是否已有工具、真正附着 | 需要权威状态判断 |
| `play_trainer` | 牌是否在手牌、基础目标选择是否合法 | 支持者次数、实际效果解析、弃牌与结算 | 训练家效果可能含隐藏信息或复杂交互 |
| `play_stadium` | 牌是否在手牌 | 是否本回合可打竞技场、替换旧竞技场 | 改变公共战场状态 |
| `use_stadium_effect` | 当前确有竞技场 UI 可用 | 是否满足发动条件、效果如何结算 | 需后端统一裁定 |
| `retreat` | 当前主动位与目标备战位参数完整 | 能量消耗、状态限制、真正撤退 | 会引起多步连锁效果 |
| `use_attack` | 攻击索引存在、按钮状态是否可点 | 最终伤害、抗性、弱点、附带效果、击倒与 Prize | 核心权威动作 |
| `use_granted_attack` | 被赋予招式的入口参数完整 | 赋予来源是否合法、招式与后续结算是否有效 | 依赖服务端效果系统 |
| `use_ability` | UI 上该特性当前看起来可用 | 是否满足发动条件、是否已用过、效果结算 | 特性经常受隐藏状态影响 |
| `end_turn` | 当前是不是自己回合的本地提示 | 真正结束回合、推进抽牌与阶段 | 不能由客户端单方面结束 |

## Choice / 交互响应

| Choice 类型 | 客户端预校验 | 服务端权威裁定 | 说明 |
| --- | --- | --- | --- |
| `mulligan_extra_draw` | 布尔参数完整 | 是否当前玩家有权选择、真正补抽 | 影响隐藏信息 |
| `setup_ready` | 本地显示引导 | 服务器确认双方 setup 交互推进 | 客户端只负责交互 UI |
| `take_prize` | 槽位索引范围检查 | 是否可拿该 Prize、真正加入手牌 | Prize 属于隐藏信息 |
| `send_out_pokemon` | 选中的备战位是否存在 | 是否允许上场、击倒替补是否合法 | 会改变真实战场 |
| `heavy_baton_target` | 目标槽位格式正确 | 是否满足道具效果条件、真正转移能量 | 必须服务端裁定 |
| `exp_share_target` | 目标槽位格式正确 | 是否满足效果条件、真正转移能量 | 必须服务端裁定 |
| `trainer_interaction` | 选择项个数、索引范围、assignment 结构格式 | 交互步骤是否合法、上下文是否匹配、真正执行效果 | 服务器是交互状态机的真实来源 |

## 客户端应承担的预校验范围

客户端应承担：

- 当前是否轮到自己操作的 UI 提示。
- 目标槽位、卡牌实例 ID、索引个数的结构化检查。
- 按钮是否可点击、输入是否完整。
- 对可见公共信息的本地推断，例如“这张牌明显不在手牌里”。
- 动画、提示、高亮、候选目标筛选。

客户端不应承担：

- 抽牌、Prize、随机投掷结果。
- 最终伤害、击倒、胜负、Prize 结算。
- 任何依赖隐藏牌库、隐藏手牌、隐藏效果状态的最终判断。
- 对局状态推进的最终提交。

## 实施建议

第一阶段建议：

1. 客户端继续做轻量预校验，减少无效请求。
2. 服务端继续保留所有真实状态变更和复杂效果结算。
3. 对每次请求携带 `request_id`、`state_seq`、`version`。
4. 如果客户端状态过期，服务端直接返回 `resync_required=true`，客户端优先重连和重同步。

第二阶段再考虑：

1. 把部分可见公共信息的可用动作推断前置到客户端。
2. 在保证可回滚的前提下做更积极的本地动画预演。
3. 视联机负载评估是否从完整快照走向增量同步。

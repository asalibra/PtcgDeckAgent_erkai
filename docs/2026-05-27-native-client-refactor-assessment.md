# PTCG Deck Agent 原生客户端迁移重构评估

## 背景与目标

本评估基于以下前提：

- Web 首访体验已被大体积 `index.wasm` / `index.pck` 明显拖慢，且这类问题并不只由带宽决定。
- 现阶段不再追求 AI 初始化能力作为首发必须项。
- 目标平台从 Web 主发转为 PC 客户端与 Android 客户端。
- 关键裁定、房间状态、重连恢复仍由服务器权威控制。
- 大量非关键逻辑、资源读取、展示交互、牌组管理可以迁移到客户端。

结论先行：

- 这不是“彻底换技术栈”，而是“保留 Godot 项目，切换主要发布形态”。
- 方向可行，而且比继续深挖 Web 首包优化更符合当前项目体量。
- 改完后体验大概率会明显变快，但不是所有服务器压力都会同比例下降。
- 通信层不必一开始就换成更复杂的专用协议；先保留 WebSocket，并补强可靠性机制，性价比最高。

## 当前架构事实

### 1. 本项目已经具备原生客户端基础

- `export_presets.cfg` 已存在 Windows Desktop 和 Android 导出预设。
- `scripts/network/UpdateChecker.gd` 已有按平台识别更新源的逻辑，说明仓库已经考虑过多平台分发。
- `scripts/autoload/GameManager.gd` 已存在桌面窗口适配逻辑，不是只为 Web 写的项目。

### 2. 联机并不是从零开始

- `scripts/network/NetworkClient.gd` 使用 WebSocket 客户端连接服务器。
- `scripts/server/ServerNetwork.gd`、`scripts/server/RoomManager.gd`、`scripts/server/GameRoom.gd` 已实现房间、会话、断线重连、消息路由。
- `scripts/network/NetProtocol.gd` 已定义客户端到服务器、服务器到客户端的消息协议。
- `scripts/server/PlayerSession.gd` 已提供 300 秒断线宽限与重连恢复。

### 3. 本地规则引擎已经成熟，可被客户端复用

- `scripts/engine/GameStateMachine.gd` 已在训练、AI、自对战、回放恢复和服务器房间中复用。
- `scenes/network/NetBattleScene.gd` 说明客户端已经可以根据服务端状态恢复并渲染本地对局视图。
- `scripts/server/ServerSerializer.gd` 已具备“按玩家视角裁剪状态”的序列化能力。

这意味着：

- 原生客户端迁移是一次“发布与职责重分配”，不是一次“引擎重写”。
- 最大收益来自资源分发与启动路径变化，而不是规则引擎本身被替换。

## 推荐目标架构

建议采用“原生胖客户端 + 权威裁定服务器”的结构，而不是完全客户端权威。

### 客户端负责

- 本地静态资源加载：卡图、音频、场景、UI 皮肤。
- 牌组管理、本地缓存、回放浏览、设置、更新检查。
- 大部分展示逻辑、交互引导、输入预校验。
- 可选的本地镜像推演：用于即时反馈、动画、按钮可用性预测。
- 单机模式、训练模式、未来的本地 AI 对战。

### 服务器负责

- 房间生命周期与匹配状态。
- 会话身份、重连恢复、超时处理。
- 最终动作合法性校验。
- 对局状态推进与关键裁定。
- 对局记录、回放、审计日志。
- 协议版本控制与最小客户端版本约束。

### 不建议放到客户端权威的部分

- 抽牌、Prize、隐藏信息的最终决定。
- 任意会改变对局真实性的 RNG。
- 回合推进、胜负判定、可执行 action 的最终判定。
- 排位、积分、正式比赛结果。

## 是否需要更换通信协议

## 结论

短期内不建议为了“可靠性”就重做成 ENet、QUIC 或自定义 UDP 协议。

当前项目是回合制卡牌对战，不是强实时动作游戏。这里最重要的不是超低延迟，而是：

- 消息顺序正确。
- 断线可恢复。
- 重发不会造成重复执行。
- 客户端和服务器版本一致。
- 发生分歧时可以强制重同步。

这些问题，WebSocket over TCP 也能解决。

## 建议方案

### Phase 1：保留 WebSocket，但补强协议可靠性

- 为每个请求增加 `request_id` / `action_id`。
- 为每个服务端状态广播增加 `state_seq` / `turn_seq`。
- 服务端对重复 action 做幂等处理，避免重连重发导致重复执行。
- 明确区分 `ack`、`error`、`resync_required`、`version_mismatch`。
- 重连后优先下发完整状态快照，再恢复 pending choice。
- 增加协议版本号与最小客户端版本校验。

对于当前项目，这一步的收益远高于切换底层传输。

### Phase 2：如果后续规模扩大，再考虑更专门的协议

只有在以下情况出现时，才值得评估原生专用协议：

- 同时在线房间显著增加，JSON 编码与完整快照广播开始浪费 CPU / 带宽。
- 需要更强的二进制压缩、增量同步、消息分类优先级。
- 未来扩展到观战、大规模赛事、复杂实时表现同步。

到那时更值得优先考虑的是：

- WebSocket 二进制帧 + 自定义结构化消息。
- 继续保留 TCP 语义，但降低 JSON 序列化成本。
- 或只在原生客户端分支评估 ENet。

现阶段不建议为了“更专业”而先把通信层重写掉。

## 工程量评估

## 结论

如果目标是“停止主打 Web，转为 Windows + Android 主发，并保留服务器权威裁定”，工程量属于中等，不是灾难级重构。

## 分档估算

### 档位 A：最小可交付迁移

目标：让朋友可安装 PC / Android 客户端，联机对战比 Web 明显稳定。

预计工作量：1 到 2 周

包含内容：

- 校验 Windows / Android 导出链路。
- 跑通联网大厅、等待房间、网络对战、结算流程。
- 修复平台相关输入、路径、权限、窗口尺寸问题。
- 关闭或降级 Web 首发地位。
- 增加版本检查与最小版本提示。

这一步已经能明显提升体验。

### 档位 B：实用版迁移

目标：把客户端职责理顺，形成稳定的原生发版流程。

预计工作量：2 到 6 周

新增内容：

- 统一安装包 / 发布包流程。
- Android 触屏 UI 与小屏适配。
- 更稳的断线重连与状态恢复。
- 协议加 `request_id`、`state_seq`、版本校验。
- 牌组、本地缓存、回放、更新提示流程补齐。
- 服务器端消息压缩与状态同步边界梳理。

这是当前最推荐的目标档位。

### 档位 C：原生优先长期形态

目标：把项目从“能跑的原生导出”升级为“持续运营的客户端产品”。

预计工作量：6 到 12 周

新增内容：

- 自动更新器。
- Windows 安装器、Android 分发渠道、签名与版本治理。
- 协议增量同步、灰度升级、回滚策略。
- 崩溃日志上报、网络诊断、自恢复流程。
- 更完整的客户端预测 / 本地镜像状态机。

这不是当前必须项，但会决定后续维护成本。

## 性能收益评估

## 1. 改完后是否一定变快

不是“数学上绝对一定”，但高概率会明显更快。

### 一定会改善的部分

- 不再依赖浏览器下载 `index.wasm` / `index.pck` 后再启动。
- 不再受浏览器缓存、service worker、Cloudflare 边缘对象影响。
- 静态资源在安装时就已落地，本地读取远快于 Web 首访拉取。
- Godot 原生运行时通常比 WebAssembly 启动链路更短。

### 不会自动消失的慢点

- 如果主场景初始化本身重，原生客户端仍然可能慢启动，只是比 Web 好。
- 如果卡牌数据库、图片缓存、回放索引加载本身太重，仍需做启动阶段拆分。
- 如果服务端仍发送完整状态快照，联机每步同步成本仍然存在，只是总体比 Web 包下载小得多。

## 2. 服务器压力会不会大大减小

结论需要拆开看：

### 会显著下降的压力

- 静态资源分发压力。
- Web 首包、CDN 缓存、浏览器兼容性相关问题。
- 客户端资源初始化对服务器带宽的占用。
- 牌组、图片、场景这类本地化后不需要反复通过网络传输的内容。

### 不会自动大幅下降的压力

- 只要服务器继续做权威裁定，它就仍然要运行核心规则引擎。
- 房间管理、断线恢复、回合合法性检查、回放记录仍在服务端。
- 活跃对局的 CPU 压力不会因为“换成客户端安装包”而直接归零。

因此，更准确的表述是：

- 带宽与静态部署压力会明显下降。
- 活跃联机对局的核心裁定 CPU 压力只会中度下降，不会消失。

## 网络稳定性评估

原生客户端方案会显著降低对带宽的敏感度，但仍然需要正确设计重连与重同步。

### 改善点

- 安装后无需反复拉大包，联机只传 action、状态快照、房间事件。
- 回合制模型天然适合较低带宽环境。
- 朋友网络有抖动时，只要会话恢复做得好，体验仍可接受。

### 仍需重点处理的点

- 移动网络切换 Wi-Fi / 4G / 5G 时的重连。
- Android 进入后台、锁屏、系统回收导致的短连中断。
- 客户端掉线后重连时的幂等与状态重同步。
- 客户端本地 UI 已预测动作，但服务器裁定失败时的回滚显示。

### 推荐的可靠性补强

- `session_token` 之外再加协议版本与客户端版本。
- action 请求加入唯一 ID。
- 服务端返回明确 `accepted` / `rejected` / `resync_required`。
- 服务端保留最近一段状态快照或日志，支持快速重同步。
- 网络错误提示从“连接断开”细化为“可自动恢复”与“需手动重进”。

## 安装与分发体验评估

## PC 客户端

优点：

- 最容易作为熟人测试版本分发。
- 可以直接发 zip 包，后续再补安装器。
- 首次安装后体验最接近传统游戏客户端。

风险：

- Windows 下若没有签名，可能触发 SmartScreen 提示。
- 没有自动更新器时，版本升级要用户重新下载。

## Android 客户端

优点：

- 相比 Web，更适合长期使用和资源本地缓存。
- 已有 Android 导出预设，可作为近期可行目标。

风险：

- 朋友需要安装 APK，首次门槛高于 Web。
- 未上架时需要允许“未知来源安装”。
- 包体可能较大，安装包下载与更新仍需要治理。

## 朋友安装会不会麻烦

结论：

- 比打开网页麻烦。
- 但对于熟人测试、小圈子联机，这个门槛通常可接受。
- 只要先把 PC 包发起来，Android 作为第二阶段推进，整体阻力不大。

## 关键坑点清单

## 1. 不要把“迁移到原生客户端”误解成“服务端可以不算规则”

只要你还需要权威裁定，就必须保留服务端核心对局推进。否则作弊、分歧和回放真实性都会出问题。

## 2. 当前网络同步是完整状态快照思路，不是增量协议

`ServerSerializer.gd` 与 `NetBattleScene.gd` 这条链路说明当前客户端主要靠服务端快照恢复状态。

这条路径的优点是简单、稳、重连方便；缺点是：

- 消息体偏大。
- 每步都反序列化、恢复状态，客户端和服务器都会有额外成本。
- 如果后续对局规模扩大，可能成为联机性能瓶颈。

因此，原生化之后不一定立刻要改，但应该列入第二阶段优化。

## 3. 版本漂移会比 Web 更敏感

Web 通常天然“刷新即最新”；原生客户端会出现：

- 客户端版本旧。
- 资源包旧。
- 协议版本旧。
- 卡牌数据版本旧。

如果不加最小版本和协议校验，后续会出现大量诡异错配问题。

## 4. Android 生命周期会暴露更多网络边缘问题

- 退到后台后 socket 被挂起。
- 切网导致连接 silently broken。
- 系统清理进程后本地恢复不完整。

这些问题不是换成 Android 就会自然处理，必须在会话恢复链路里补齐。

## 5. 包体问题从“网页首包”变成“安装包与更新包”

Web 慢的问题不会凭空消失，只是位置变了：

- 原来是用户打开网页很慢。
- 迁移后会变成 APK / 桌面包体偏大、首次下载和升级偏慢。

所以资源治理仍然要做，只是优先级从“首访”变成“安装与更新”。

## 6. 如果客户端承担更多本地预测，要处理回滚体验

客户端为了交互流畅，可以在本地预判某些按钮可点、动画可播，但最终仍要以服务端裁定为准。

这要求：

- 预测结果不得直接写成最终事实。
- 服务端拒绝时 UI 能回滚。
- 所有关键随机和隐藏信息必须以后端为准。

## 推荐实施路线

## 第一阶段：停止把 Web 当主交付物

- 保留 Web 版作为演示入口或轻量试玩入口。
- 把正式可玩版本转向 Windows 客户端。
- 同步保留 Android 导出路径，但不把 Android 首发压力放到第一周。

## 第二阶段：稳定原生联机主链路

- 先沿用现有 WebSocket 协议。
- 增加 action 唯一 ID、状态序号、版本校验、重同步消息。
- 完整跑通大厅、等待房间、对战、断线、结算。

## 第三阶段：再评估协议与同步粒度

- 如果快照广播已经足够稳定，就先不动。
- 如果发现服务端或客户端在频繁序列化 / 恢复整局状态上开销过高，再评估增量同步。
- 不要把协议重写排在发版前面。

## 最终判断

对于当前项目，最推荐的路线是：

- 保留 Godot。
- 保留服务端权威裁定。
- 主发布形态切到 Windows + Android。
- 大量展示与非关键逻辑下放到客户端。
- 通信层先继续使用 WebSocket，并补齐可靠性设计，而不是立即换更复杂的专用协议。

这条路线能同时解决当前 Web 首包过重、缓存链路复杂、朋友使用体验差的问题，而且工程量可控。

## 建议的立即行动项

1. 先做一次联机链路切换评审，明确哪些 action 必须继续由服务端裁定，哪些只做客户端预校验。
2. 给现有协议补 `request_id`、`state_seq`、`version`、`resync_required` 四个核心字段。
3. 用 Windows 导出做第一版可分发包，优先验证真实安装与联网体验。
4. 为 Android 单独列出后台切回、锁屏恢复、切网重连的专项测试清单。
5. 把 Web 版降为演示入口，不再把核心体验押在浏览器首包上。
